# objdiff

Object file diffing and match validation via encounter/objdiff ‚Äî comparing
compiled object files against original assembly targets for matching
decompilation projects.

> Used by Build Gates in `n64-decomp`, `windows-game-decomp`, and
> `xboxrecomp` to verify function-level matching before claiming success.
> objdiff is the final proof gate ‚Äî no match claim is valid without it.

## What it enables

- List all units from `objdiff.json` with per-function match status
- Diff a specific object: compare target (original .o) vs base (recompiled .o)
- Generate progress reports with match percentages for project tracking
- Automated match validation ‚Äî no manual asm comparison needed
- Supports ARM, ARM64, MIPS, PowerPC, SuperH, and x86 object formats

## When to use

Use this tool for:
- Verifying recompiled C produces identical assembly to original binary
- Tracking match progress across a project (total matched / total functions)
- Build Gate validation ‚Äî objdiff clean must pass before claiming match
- Validating compiler + flag changes against the original output

Do not use it for:
- Runtime behavior validation (use `cdb-debug`, `bizhawk`, or platform bridge)
- Source-level comparison (use `read_file` on generated C)
- Projects without `objdiff.json` config ‚Äî generate it first (usually via
  `configure.py` or project build script)
- Single-shot scratch testing (use decomp.me for quick experiments)

## Boot / Connection Check

1. Verify objdiff MCP server is running:
   - `/mcp tools objdiff` ‚Äî lists all available tools
   - Server shows `Connected` on the splash screen or `/mcp` output
2. Verify `objdiff.json` exists in the project root:
   - `read_file objdiff.json` (first 20 lines) ‚Äî confirms config is present
   - If missing: generate via `configure.py` or project build script
3. Verify the build system is configured:
   - `custom_make` field in `objdiff.json` (e.g., `ninja` or `make`)
   - Project builds successfully before diffing
4. If unavailable:
   - Check `/mcp status objdiff` for connection state
   - Install: `cargo install --locked --git https://github.com/encounter/objdiff.git objdiff-cli`
   - Ensure `objdiff-mcp` is on PATH or `RECOMPHAMR_MCP_OBJDIFF_COMMAND` is set

## Setup

1. Install objdiff-cli:
   `cargo install --locked --git https://github.com/encounter/objdiff.git objdiff-cli`
2. Ensure `objdiff-mcp` is on PATH (or set `RECOMPHAMR_MCP_OBJDIFF_COMMAND`)
3. Project must have `objdiff.json` config (generated by `configure.py` or manually)
4. Start recomphamr ‚Äî connect with `/mcp connect objdiff`
5. Load `/skill objdiff` ‚Äî unlocks `objdiff.*` tools
6. Verify: `/mcp tools objdiff`

## Evidence Protocol

Every diff result should be classified:

| Result | Meaning | Action |
|---|---|---|
| **MATCH** | Object file is byte-identical | Record as CONFIRMED match; update match log |
| **MISMATCH** | Object file differs | Record divergence; check compiler/flags/asm |
| **BUILD_FAIL** | Base object failed to build | Fix build error before re-diffing |
| **MISSING** | Object not in config or not built | Verify unit entry in `objdiff.json` |

Record per-function status in:
- `docs/match_log.md` ‚Äî per-function match results with compiler/flags
- `REPHAMR_STATE.md` ‚Äî overall match percentage, phase, active diffs

## Common Operations

| Operation | Tool Call | Output | Notes |
|---|---|---|---|
| List all units | `objdiff.list_units` | Unit list with status | Overview of project state |
| Diff one unit | `objdiff.diff_unit` | Match/Mismatch + details | Specify unit name or path |
| Generate report | `objdiff.report` | Match % + per-unit status | For progress tracking |
| Watch for changes | `objdiff.watch` | Auto re-diff on rebuild | Long-running validation |

Build system note: `objdiff` uses the project's `custom_make` (e.g., `ninja`)
to build base objects before diffing. Ensure the build system is configured
and working before diffing.

## Guardrails

1. **objdiff is the final proof gate.** Never claim a match without a clean
   objdiff result. "Looks identical" is not sufficient.
2. Single-function objdiff match does not prove the compiler/flags are correct
   for the entire project ‚Äî test on multiple functions across subsystems.
3. objdiff relies on `custom_make` to build base objects ‚Äî if the build
   system is broken, objdiff results are invalid.
4. Target objects must be built from the original binary (not from a previous
   build) ‚Äî stale target objects produce false positives.
5. Match percentage is a progress metric, not a quality metric ‚Äî a 90% match
   with 10% unverified is not a "done" project.
6. Build and diff one function at a time during Phase 2 (first match) ‚Äî bulk
   diffs before confirming compiler/flags waste time.

## Failure Handling

| Failure | Meaning | Next Step |
|---|---|---|
| `objdiff-mcp` not on PATH | Server binary missing | `cargo install objdiff-cli`; set `RECOMPHAMR_MCP_OBJDIFF_COMMAND` |
| `objdiff.json` not found | Project not configured for objdiff | Generate via `configure.py` or project build script |
| Diff returns MISMATCH | Function doesn't match | Check compiler version/flags; verify asm is from latest split |
| Diff returns BUILD_FAIL | Base object build failed | Fix build error; verify `custom_make` in `objdiff.json` |
| Unit not in list | `objdiff.json` missing the function | Add unit entry: name, target_path, base_path |
| Server Disconnected | objdiff-mcp not running | Reconnect; verify binary on PATH |

## Session Close

1. Update `docs/match_log.md` with objdiff results per function.
2. Update `REPHAMR_STATE.md` with match percentage, confirmed matches, active mismatches.
3. Report: functions diffed, matches confirmed, mismatches with likely cause, next functions to match.
